## Status of This Document

This document specifies an open standard for the Basalt DSL. Distribution of this document is unlimited. Comments and proposals should be submitted to `github.com/DiegoDev2/basalt/spec`.

---

## Abstract

This RFC defines the Basalt DSL — a declarative language for describing backend API services. The language is designed to be:

- **Human-readable** — writable by developers without tooling
- **Machine-readable** — parseable by any compiler or generator
- **AI-friendly** — simple enough that any AI model can generate valid files from natural language descriptions

A Basalt file (`.bs`) describes the **intent** of a backend. A Basalt-compatible generator is responsible for the **implementation**. This separation allows the same `.bs` file to target different frameworks, languages, and databases.

---

## 1. Motivation

Current AI-assisted backend development requires models to generate implementation code directly. This approach has three fundamental problems:

**1.1 Token inefficiency.** Generating 200 lines of TypeScript costs significantly more tokens than generating 20 lines of a structured DSL.

**1.2 Inconsistency.** Each generation produces structurally different code, making maintenance difficult.

**1.3 Hallucination surface.** The larger and more complex the output format, the more likely a model is to produce invalid or insecure code.

Basalt addresses all three by constraining the AI output to a small, well-defined grammar. The AI generates `.bs`. The generator produces code.

---

## 2. Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.

- **DSL** — Domain-Specific Language
- **Generator** — A tool that compiles `.bs` files into executable code
- **Consumer** — An AI model or human author producing `.bs` files
- **Host** — The runtime environment where generated code executes

## 3. Specification

### 3.1 Encoding

A `.bs` file MUST be encoded as UTF-8. Line endings MAY be LF or CRLF. The file MUST end with a newline character.

### 3.2 Comments

Comments begin with `#` and extend to the end of the line. Comments MAY appear anywhere whitespace is allowed.

```
# This is a comment
config {
  db: supabase  # inline comment
}
```

### 3.3 Block Structure

All top-level constructs are blocks. A block consists of a keyword, an optional name, and a body enclosed in `{}`.

```
keyword ["name"] {
  key: value
  ...
}
```

### 3.4 Block Types

A valid `.bs` file MUST contain:
- Exactly one `config` block
- At least one `table` block
- At least one `resource` block

A valid `.bs` file MAY contain:
- Zero or more `role` blocks

### 3.5 Config Block

```
config {
  db:         <provider>
  auth:       <method>
  framework:  <framework>   # OPTIONAL
  lang:       <language>    # OPTIONAL
  pagination: <boolean>     # OPTIONAL
}
```

**db** (REQUIRED) — Database provider. MUST be one of: `supabase`, `postgres`, `mysql`.

**auth** (REQUIRED) — Authentication method. MUST be one of: `jwt`, `apikey`, `none`.

**framework** (OPTIONAL) — Target framework. MUST be one of: `express`, `fastify`, `hono`. Default: `express`.

**lang** (OPTIONAL) — Output language. MUST be one of: `typescript`, `javascript`. Default: `typescript`.

**pagination** (OPTIONAL) — Enable automatic pagination on list endpoints. MUST be `true` or `false`. Default: `false`.

### 3.6 Table Block

```
table "<name>" {
  <field_name>: <type> [decorators...]
  ...
}
```

Table names MUST be lowercase, and MUST use underscores for word separation. Each table MUST include a field with `@primary`. It is RECOMMENDED that every table include `id: uuid @primary` and `created_at: timestamp @default(now)`.

#### 3.6.1 Types

| Keyword | Description |
|---------|-------------|
| `uuid` | Universally unique identifier |
| `string` | Variable-length text |
| `number` | Integer number |
| `float` | Floating-point number |
| `boolean` | True or false value |
| `timestamp` | Date and time with timezone |
| `json` | Arbitrary JSON object |

#### 3.6.2 Structural Decorators

| Decorator | Description |
|-----------|-------------|
| `@primary` | Marks field as primary key. MUST appear exactly once per table. |
| `@unique` | Adds a unique constraint. |
| `@nullable` | Allows null values. Fields are NOT NULL by default. |
| `@default(value)` | Sets a default value. `value` MAY be a string, number, boolean, or `now`. |
| `@ref(table.field)` | Declares a foreign key. The referenced table and field MUST exist in the same file. |

#### 3.6.3 Validation Decorators

Validation decorators generate server-side validation logic. Generators MUST enforce these constraints in the generated handlers.

| Decorator | Applies To | Description |
|-----------|-----------|-------------|
| `@min(n)` | `string` | Minimum character length |
| `@min(n)` | `number`, `float` | Minimum numeric value |
| `@max(n)` | `string` | Maximum character length |
| `@max(n)` | `number`, `float` | Maximum numeric value |
| `@regex("pattern")` | `string` | Value must match the given regular expression |
| `@email` | `string` | Value must be a valid RFC 5322 email address |
| `@url` | `string` | Value must be a valid URL |

Multiple decorators MAY be applied to a single field. Decorators are evaluated in order.

### 3.7 Role Block

```
role "<name>" {
  description: "<text>"
}
```

Roles define named permission groups. Role names MUST be lowercase. The `description` field is OPTIONAL but RECOMMENDED.

### 3.8 Resource Block

```
resource "<name>" {
  table:    <table_name>
  db:       <provider>    # OPTIONAL — overrides config.db
  paginate: <boolean>     # OPTIONAL — overrides config.pagination

  endpoints {
    <METHOD> <path>  roles: [<role>, ...]
    ...
  }
}
```

The `table` field MUST reference a table defined in the same file.

#### 3.8.1 Endpoints

Each endpoint consists of an HTTP method, a path, and an optional `roles` list.

```
GET    /users              roles: [guest, user, admin]
POST   /users              roles: [admin]
GET    /users/:id          roles: [user, admin]
PUT    /users/:id          roles: [user, admin]
DELETE /users/:id          roles: [admin]
```

**Methods** — MUST be one of: `GET`, `POST`, `PUT`, `PATCH`, `DELETE`.

**Paths** — MUST begin with `/` followed by the resource name. Path parameters are prefixed with `:`.

**Roles** — When specified, MUST reference roles declared in a `role` block. When omitted, access control follows the global `auth` config.

#### 3.8.2 Automatic Pagination

When pagination is enabled, `GET` list endpoints MUST support the following query parameters:

| Parameter | Type | Default | Max |
|-----------|------|---------|-----|
| `page` | integer | 1 | — |
| `limit` | integer | 20 | 100 |

Generators MUST produce responses in this format:

```json
{
  "data": [],
  "pagination": {
    "page": 1,
    "limit": 20,
    "total": 0,
    "pages": 0
  }
}
```

#### 3.8.3 Multiple Databases

A resource MAY specify a `db` field to override the global database provider. Generators MUST initialize a separate database client for each unique `db` value used across resources.

## 4. Validation Rules

A Basalt generator MUST reject any `.bs` file that violates the following rules, and MUST produce a descriptive error message including the file name, line number, and column number.

| Rule | Error |
|------|-------|
| More than one `config` block | `E001: duplicate config block` |
| `resource.table` references undefined table | `E002: unknown table "<name>"` |
| `@ref` references undefined table | `E003: unknown ref target "<table.field>"` |
| Endpoint role references undefined role | `E004: unknown role "<name>"` |
| Table has no `@primary` field | `E005: table "<name>" has no primary key` |
| Duplicate field name in table | `E006: duplicate field "<name>" in table "<table>"` |
| Invalid type keyword | `E007: unknown type "<type>"` |
| Invalid HTTP method | `E008: unknown method "<method>"` |
| Duplicate endpoint in resource | `E009: duplicate endpoint "<method> <path>"` |

---

## 5. Conformance

A **Basalt-conformant generator** MUST:

- Parse all constructs defined in this RFC
- Reject files that violate Section 4 rules
- Produce error messages with file, line, and column
- Generate separate code artifacts for each resource
- Enforce all validation decorators in generated handler code
- Support all HTTP methods defined in Section 3.8.1

A **Basalt-conformant generator** MAY:

- Support additional decorator extensions prefixed with `@x-`
- Generate additional files beyond the minimum required set
- Support additional database providers, frameworks, or languages beyond those listed

---

## 6. Security Considerations

**6.1 Password fields.** Any field named `password` SHOULD have `@min(8)` applied. Generators SHOULD warn if this is not the case.

**6.2 Role enforcement.** Generators MUST generate role-checking middleware that runs before handlers. A request from an unauthenticated user to an endpoint with no `guest` role MUST be rejected with HTTP 401.

**6.3 Validation.** Generators MUST enforce all validation decorators server-side. Client-side validation is OPTIONAL.

**6.4 SQL injection.** Generators MUST use parameterized queries or an ORM layer. String interpolation in SQL queries is PROHIBITED.
